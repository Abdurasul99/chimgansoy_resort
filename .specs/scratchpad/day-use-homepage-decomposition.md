# Phase 4 scratchpad — day-use homepage decomposition

## Входы и ограничения

- Task: `.specs/tasks/draft/redesign-day-use-homepage.feature.md`.
- Codebase analysis: `.specs/analysis/day-use-homepage-codebase.md`.
- Источник бизнес-истины: `AGENTS.md` и явное подтверждение владельца перед реализацией.
- Сохраняются: App Router, RU/UZ/EN, GET-переход на `/[locale]/bron`, Exely context/container/failure state, разделение `booking`/`inquiry`, данные и страницы размещения.
- Убираются из публичной главной: intro, ручной переключатель сезона, `/nomera`, rooms/master-plan и противоречащие day-use FAQ/SEO.
- Не входят в критический путь: новая фотосъёмка и необязательная аналитика. Для релиза используются уже одобренные `gal*`-фото; AC-28 проверяется только при готовой среде аналитики, иначе нужен зафиксированный waiver владельца.

## Решения декомпозиции

1. Ограничить реализацию девятью шагами, каждый не крупнее L.
2. Отделить визуальную работу от Exely-gate: интерфейс можно собирать параллельно, но AC-07/AC-08 нельзя закрыть без проверки реального single-date query-контракта.
3. Для AC-07 нужен единый клиентский state дневного запроса, используемый hero, ценовым и финальным CTA. Не полагаться на дублирование `checkout` или неподтверждённые query-ключи.
4. Intro удаляется одним атомарным изменением: head gate, mount/import, CSS, переводы и компонент.
5. Ночёвки скрываются, но `rooms.ts`, `/nomera/[slug]`, Exely IDs и assets не удаляются.
6. SEO schema меняется на подтверждённый day-use тип; при отсутствии одобрения спорный lodging schema удаляется.

## Оценочная шкала

- **S**: до половины инженерного дня, один локальный контракт.
- **M**: примерно один инженерный день, несколько связанных файлов.
- **L**: два–три инженерных дня с тестами и ручной проверкой. Всё, что выходит за L, должно быть снова разделено.
